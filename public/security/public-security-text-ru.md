# Публичный текст для сайта / whitepaper

## Вариант короткий

Atlas System разделяет два уровня безопасности: защиту кода от внешних атак и архитектурные полномочия системы.

Пользовательские операции проверяются на уровне смарт-контракта: claim доступен только владельцу соответствующего ордера, ключевые пользовательские функции используют reentrancy-защиту, а операции с токенами выполняются через SafeERC20.

Отдельно раскрываются административные функции, включая Transport-модуль, treasury-настройки, комиссии и управление LP-позицией. Эти элементы относятся не к внешнему взлому, а к архитектурным и управленческим рискам, которые должны быть понятны участникам заранее.

## Вариант для блока “Security Review”

Atlas System проходит внутренний security review, в котором отдельно проверяются:

- защита пользовательских claim-операций;
- невозможность claim чужого ордера;
- reentrancy-риски;
- корректность расчёта reward и fee;
- Transport-логика;
- owner-полномочия;
- зависимость от LP-позиции и внешней инфраструктуры Pancake V3.

Мы не смешиваем два разных вопроса: может ли посторонний адрес атаковать контракт, и какие полномочия есть у владельца системы. Это разные типы риска, и они раскрываются отдельно.

## Вариант для FAQ

### Может ли посторонний пользователь вывести чужие средства?

Пользовательские claim-операции защищены проверкой владельца ордера. Это означает, что адрес, который не является владельцем конкретного ордера, не может вызвать claim по этому ордеру.

Перед публичным запуском эта логика должна быть дополнительно проверена через автоматический анализ, fuzzing, инвариантные тесты и тестнет-испытание.

### Есть ли у системы административные полномочия?

Да. В архитектуре есть owner-функции и Transport-модуль. Они нужны для управления отдельными параметрами системы и исполнения определённых процессов, но относятся к управленческим рискам. Эти полномочия должны быть раскрыты отдельно от темы внешнего взлома.

### Можно ли говорить, что контракт audited?

Нет, пока не проведён внешний аудит. Корректная формулировка:

```text
Internal Security Review
```

или:

```text
Security Review in progress
```

## Что нельзя писать

Не использовать:

```text
Audited
```

```text
100% secure
```

```text
Funds are guaranteed
```

```text
Impossible to hack
```

## Что можно писать

Можно использовать:

```text
Internal Security Review
```

```text
Access control, reentrancy and claim logic are being tested separately from architectural owner permissions.
```

```text
Atlas System separates external attack resistance from governance and operational risks.
```
