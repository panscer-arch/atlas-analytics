# Atlas System — Foundry stress-test 1000 users

Версия: V1  
Дата: 2026-06-05  
Статус: пройден локально в mock-окружении

## Что проверяли

Целью было закрыть большой локальный stress-сценарий, который раньше был указан как следующий этап:

```text
1000 пользователей
50000 lockup
100000 claim-попыток
```

Тестовый файл:

```text
test/AtlasStress.t.sol
```

Команда:

```text
forge test --match-contract AtlasStressTest -vv --gas-limit 10000000000
```

## Результат

```text
Ran 1 test for test/AtlasStress.t.sol:AtlasStressTest
[PASS] testStress1000Users50000Lockups100000ClaimAttempts()
Suite result: ok. 1 passed; 0 failed; 0 skipped
```

## Что именно делал тест

1. Создал 1000 разных адресов пользователей.
2. Для каждого пользователя создал по 50 Lockup-ордеров.
3. Всего было создано 50000 Lockup-ордеров.
4. Проверил, что `nextOrderId == 50000`.
5. Перевел время вперед после lockup-периода.
6. Выполнил 50000 успешных claim по владельцам ордеров.
7. Выполнил 50000 повторных claim-попыток по тем же ордерам.
8. Повторные claim были остановлены ожидаемым revert:

```text
Order already claimed or not exist
```

Итого:

```text
100000 claim-попыток
```

## Почему gas metering был отключен

Первый прогон без отключения gas-accounting уперся в лимит выполнения Foundry:

```text
~1.07 млрд gas: revert
10 млрд gas: revert
```

После этого в stress-тесте был использован `pauseGasMetering()`, чтобы проверить именно логическое поведение массового сценария, а не стоимость исполнения одной гигантской тестовой транзакции.

Это важно: данный тест является логическим stress-test в mock-окружении, а не gas benchmark и не production-performance benchmark.

## Перевод на человеческий язык

В локальной симуляции контракт выдержал сценарий:

```text
1000 пользователей создали 50000 ордеров,
затем было сделано 100000 попыток claim.
```

Первая волна claim прошла по владельцам ордеров. Вторая волна повторных claim не прошла, потому что контракт уже пометил ордера как закрытые.

Это подтверждает важную пользовательскую логику:

```text
Lockup-ордер нельзя вывести повторно даже при массовом количестве операций.
```

## Что это не доказывает

- Это не testnet battle test.
- Это не внешний аудит.
- Это не проверка реальной Pancake V3 ликвидности.
- Это не gas/performance benchmark.
- Это не доказывает, что Daily Flow и Transport выдержали такой же массовый сценарий.
