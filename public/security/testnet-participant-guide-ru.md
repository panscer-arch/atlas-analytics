# Atlas System — инструкция участника BNB Testnet Battle

Версия: V1  
Дата: 2026-06-05  
Статус: инструкция подготовлена, публичное испытание еще не начато

## Цель испытания

Проверить testnet-версию Atlas System на воспроизводимые security-сценарии. Участник не должен доказывать теорию словами: нужен конкретный сценарий, транзакции и понятное объяснение, почему это exploit или риск для пользователей.

## Что нужно участнику

1. Wallet в BNB Smart Chain Testnet.
2. Test BNB для gas.
3. Test token / MockUSDT от команды.
4. Contract registry с адресами контрактов.
5. ABI или verified source.

## Что проверять

- Может ли чужой адрес сделать claim за пользователя.
- Можно ли повторно вывести один и тот же lockup-ордер.
- Можно ли вывести больше, чем разрешено правилами.
- Можно ли обойти owner-only ограничения.
- Можно ли нарушить учет reward/fee.
- Можно ли заблокировать claim для группы пользователей.
- Есть ли проблема в LP / liquidity сценарии.

## Что считается валидным report

Report считается валидным, если есть:

1. Название проблемы.
2. Уровень критичности.
3. Адреса контрактов.
4. Tx hashes.
5. Пошаговое воспроизведение.
6. Ожидаемое поведение.
7. Фактическое поведение.
8. Объяснение влияния на средства, права или доступность.

## Что не считается exploit

- Ошибка интерфейса без влияния на контракт.
- Теоретическое предположение без транзакций.
- Поведение, заранее описанное в правилах.
- Попытка атаковать не testnet-версию.
- Report без воспроизводимых шагов.

## Severity

Critical:

- вывод чужих средств;
- вывод больше разрешенного;
- обход owner-only;
- повторный claim;
- массовая блокировка claim.

High:

- ошибка учета, влияющая на balance/reward/fee;
- DoS для группы пользователей;
- обход важной проверки.

Medium:

- некорректный revert-path;
- сценарий с ограниченным влиянием;
- ошибка, требующая редких условий.

Low:

- документационная ошибка;
- ABI/UI confusion;
- улучшение сообщений об ошибках.
