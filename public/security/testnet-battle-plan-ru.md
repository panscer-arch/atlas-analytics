# Atlas System — план BNB Testnet Battle Test

Версия: V1 draft  
Дата: 2026-06-05  
Статус: план испытания, само испытание еще не проведено

## Цель

Показать, что код проверяется не только командой, но и внешними участниками в воспроизводимых testnet-сценариях.

Главный вопрос:

```text
Может ли внешний пользователь найти воспроизводимый способ вывести больше, чем разрешено логикой контракта, вызвать чужой claim или обойти owner-only ограничения?
```

## Что проверяем

### Пользовательские сценарии

- Нельзя сделать claim по чужому Lockup-ордеру.
- Нельзя сделать claim по чужому Daily-ордеру.
- Нельзя повторно вывести уже закрытый Lockup-ордер.
- Нельзя вывести больше, чем доступно по правилам выбранного продукта.
- Нельзя вызвать Transport.claimReferral без owner-доступа.

### Экономика и учет

- Корректный расчет Lockup reward.
- Корректный расчет Daily reward по дням.
- Корректное удержание platform fee.
- Корректная обработка недостаточной ликвидности.

### LP / Pancake V3 окружение

- Поведение при изменении tick.
- Поведение при маленькой ликвидности.
- Поведение при большом количестве lockup/claim.
- Поведение при смене tokenId владельцем.

## Формат испытания

1. Развернуть финальные testnet-контракты на BNB Testnet.
2. Опубликовать адреса контрактов, ABI, исходный код и инструкцию.
3. Выдать участникам testnet USDT/mock token.
4. Открыть период тестирования на 3-7 дней.
5. Принимать отчеты только с воспроизводимым сценарием:
   - шаги атаки;
   - адреса транзакций;
   - ожидаемое поведение;
   - фактическое поведение;
   - почему это exploit.

## Минимальный bounty-подход

Критический уровень:

- вывод чужих средств;
- обход owner-only;
- повторный claim;
- вывод больше разрешенной суммы;
- остановка claim для всех пользователей через воспроизводимый сценарий.

Средний уровень:

- ошибка учета;
- некорректный расчет fee/reward;
- сценарий DoS для отдельной операции.

Низкий уровень:

- UI/ABI confusion;
- неполные события;
- неточные revert-сообщения;
- документационные ошибки.

## Что публиковать после теста

- Количество участников.
- Количество прогнанных сценариев.
- Количество найденных проблем по уровням.
- Какие проблемы исправлены.
- Какие проблемы признаны архитектурными рисками.
- Ссылку на финальный отчет.

## Что нельзя писать до проведения

- Нельзя писать, что battle test проведен.
- Нельзя писать, что внешние участники подтвердили безопасность.
- Нельзя писать `battle-tested`, если публичный тест еще не был завершен.

## Корректная публичная формулировка сейчас

BNB Testnet Battle Test запланирован как следующий этап проверки Atlas System. Его цель — дать внешним участникам возможность воспроизвести потенциальные exploit-сценарии в testnet-среде и получить вознаграждение за подтвержденные критические находки.
