# Atlas Telegram Tasks Bot

Бот нужен как intake-слой для Telegram-задач. Он не является AI-помощником и не отвечает по базе знаний.

Голос бота: он общается как `Суперсус` — дружелюбно, чуть комично, коротко и по делу. В интерактивных командах и ATL/YouTrack-уведомлениях он пишет как живой операционный участник: что двигается, что ушло на тест, где нужен ответ и что надо подпнуть. Обычные ответы не начинаются с `Суперсус:`. Юмор допускается в стиле рабочего товарища с лёгким сухим абсурдом, но без копирования известных персонажей, национальных карикатур и токсичности. Чистый endpoint `/api/telegram/send-message` отправляет текст как есть, без автоперсонажа.

## Что умеет

- создавать задачи из Telegram в досках аналитики;
- принимать задачи только по явной команде `/task` или реакции `💋`;
- использовать forwarded/reply/voice как контекст для `/task`, не спрашивая категорию автоматически;
- распознавать голосовые сообщения, когда на них ответили командой `/task`;
- сохранять решения, вопросы, отчёты и напоминания;
- вести память Telegram-чата: участники, стиль общения, внутренние шутки, последние контексты;
- тихо синхронизировать важные куски этой памяти в Hermes;
- отвечать на `/sus`, `/sus стиль`, `/sus память`, чтобы команда видела его режим и состояние памяти;
- читать прямые обращения `Суперсус`, `SuperSUS`, `Кодекс`, `Codex` и отвечать через Hermes без отдельной команды;
- показывать активные задачи по категории;
- отправлять план на день в общий чат по подзадачам, ответственным, статусам и дедлайнам;
- писать в те же content keys, которые читает сайт.

```text
/sus кто тормозит
/sus пни

/sus дайджест
/sus релиз
/sus разложи
/sus настроение
/sus дневник
/sus кто есть кто
/sus биография
/sus день
/sus история
/task marketing Сделать 10 картинок для FB
/task launch @rubi до 01.06 Проверить ссылку TikTok в закрепе
/task daily Созвониться по запуску
/today

/report FB создан, ссылка в закрепе, жду правки по КП

```

Гермес:

```text
/hermes запомни решение: дневник в Суперсусе ведём каждый день
гермес разложи последние задачи по приоритетам
```

Команда `/hermes` отправляет сообщение в Hermes Agent на VPS через локальный мост `hermes-telegram-bridge.service`.
Текущий Telegram-бот остаётся единственным процессом, который читает Telegram updates, поэтому команды задач и голосовые не ломаются.

Также можно обращаться без команды:

```text
Суперсус, что думаешь по этой задаче?
Кодекс, прочитай выше и скажи что делать
```

Бот читает обращение, берёт свежий контекст чата из памяти и отправляет вопрос в Hermes. Ответ приходит reply на исходное сообщение. Это не создаёт задачу автоматически.

Reply-сценарий:

```text
Ответить на сообщение командой:
/task marketing
```

Бот возьмёт текст исходного сообщения как задачу.

Голосовые:

1. Отправить голосовое в чат.
2. Бот сохранит его как контекст и не будет сам спрашивать категорию.
3. Если это задача, ответить на голосовое командой `/task category`.

Также можно ответить на голосовое командой:

```text
/task marketing
```

Тогда бот распознает voice-message и сразу поставит задачу в маркетинг.

Обычные reply/forward/voice сообщения не создают задачу и не вызывают вопрос `Куда поставить задачу?`.

Шутливый reply на вопрос Суперсуса про категорию:

```text
в угол
```

Бот не создаёт задачу, ставит лёгкую реакцию и отвечает шуткой именно reply на сообщение.

## Память Суперсуса

Память чата хранится отдельно от задач:

```text
atlas.analytics.telegramChatMemory.v1
```

Что запоминается:

- участники чата и как они обычно общаются;
- короткая история последних сообщений;
- прямые обращения к `Суперсусу`, `SuperSUS`, `Кодексу`, `Codex`;
- внутренние шутки, например `в угол`;
- кто как подкалывает Суперсуса;
- какие ответы лучше давать коротко, reply-кнопкой и без канцелярита.

Обычные сообщения не становятся задачами. Задачи создаются только через `/task`, реакцию `💋` или явный текстовый триггер.

Когда появляется важный контекст или накопилась пачка сообщений, бот тихо отправляет memory note в Hermes через `hermes-telegram-bridge.service`. В Telegram при этом ничего не пишет.

Команды самого Суперсуса:

```text
/sus
/sus стиль
/sus память
```

`/sus` показывает, что он умеет. `/sus стиль` показывает границы характера и юмора. `/sus память` показывает количество участников, шуток и время последней синхронизации с Hermes без раскрытия сырого чата.

Операционные режимы:

- `/sus что горит` — самые срочные места: блокеры, вопросы, зависшие задачи;
- `/sus зависло` — задачи без движения больше 24 часов;
- `/sus тест` — задачи, которые ждут проверки;
- `/sus кто тормозит` — где ждём человека, ответ или действие, без токсичного поиска виноватых;
- `/sus пни` — мягкий список кого/что вернуть в движение;
- `/sus решения` — последние решения из `/decision`;
- `/sus дайджест` — короткая сводка по задачам, вопросам и решениям;
- `/sus релиз` — быстрый health-check API и монитора задач;
- `/sus разложи` — разложить reply или текст на список задач без автосоздания.
- `/sus настроение` — живое состояние Суперсуса по текущим задачам, хвостам и времени суток;
- `/sus дневник` — создать запись в дневнике Суперсуса и показать последние записи;
- `/sus кто есть кто` — рабочая память по участникам чата: сообщения, заметки, фразы, без выдуманных биографий;
- `/sus биография` — персонажная биография Суперсуса;
- `/sus день` — как он живёт, ест, спит и делает технические перерывы;
- `/sus история` — короткий курьёзный случай из жизни задач.

Биография и бытовые привычки Суперсуса — это lore Telegram-бота, а не утверждение, что он физический человек. В чате он может поддерживать беседу как живой персонаж, но на прямой вопрос отвечает честно.
Для живости у Суперсуса есть банк фраз по настроению, дневнику, ночному режиму, тесту, зависшим задачам и деплою. Последние использованные фразы сохраняются в памяти чата, чтобы он не повторялся подряд.

Посмотреть память:

```bash
curl -H "x-telegram-memory-token: $TELEGRAM_MEMORY_READ_TOKEN" \
  https://supersussystem.com/api/telegram/memory
```

Посмотреть память конкретного чата:

```bash
curl -H "x-telegram-memory-token: $TELEGRAM_MEMORY_READ_TOKEN" \
  'https://supersussystem.com/api/telegram/memory?chatId=-5158247269'
```

Сырой content-key памяти закрыт от общего `/api/content/<key>`, чтобы реальные фразы из чата не торчали наружу.

Реакция `💋`:

1. Написать обычное сообщение с задачей в чат.
2. Поставить на него реакцию `💋`.
3. Бот спросит: `задача поймана. Куда кладём, мой друг?`
4. Выбрать категорию кнопкой.

Если сообщение было отправлено до перезапуска бота и его нет в памяти, бот попросит переслать сообщение заново или ответить на него командой `/task`.

Управление задачами:

```text
/my
/my @username
```

Показывает активные задачи по ответственному.

Reply на исходное сообщение, из которого была создана задача:

```text
/assign @username
/status Готово
/deadline 05.06
/done
```

Также можно поставить реакцию `✅` на исходное сообщение задачи, чтобы закрыть её.

Forward-сценарий:

1. Переслать сообщение в чат.
2. Бот сохранит forward как контекст и не будет сам спрашивать категорию.
3. Если это задача, ответить на forward командой `/task category` или поставить реакцию `💋`.
4. После выбора категории задача появится в аналитике.

## Настройка на сервере

Создать файл:

```bash
sudo nano /etc/atlas-telegram-bot.env
```

Минимально:

```bash
TELEGRAM_BOT_TOKEN=123456:telegram-token
OPENAI_API_KEY=sk-...
```

Опционально ограничить чаты:

```bash
TELEGRAM_ALLOWED_CHAT_IDS=-1001111111111,-1002222222222
TELEGRAM_PUSH_CHAT_ID=-1001111111111
ATLAS_YOUTRACK_TELEGRAM_CHAT_ID=-1001111111111
TELEGRAM_MEMORY_READ_TOKEN=long-random-token
OPENAI_TRANSCRIPTION_MODEL=whisper-1
TELEGRAM_TRANSCRIBE_MAX_BYTES=26214400
```

`TELEGRAM_PUSH_CHAT_ID` — чат, куда кнопка `Push` из подзадачи отправляет задачу команде. Если не задан, используется первый чат из `TELEGRAM_ALLOWED_CHAT_IDS`.
`ATLAS_YOUTRACK_TELEGRAM_CHAT_ID` — чат для автоматических уведомлений ATL task monitor по новым задачам, комментариям, смене статуса и исполнителя. Если не задан, используется `TELEGRAM_PUSH_CHAT_ID` или первый доступный чат.

Команда `/atl` показывает текущую сводку задач без записи snapshot, чтобы ручная проверка не съедала автоматические уведомления. Серверный монитор вызывает `/api/youtrack/check` с `notify: true` раз в 10 минут и сохраняет snapshot после успешной обработки.

Автоматические ATL/YouTrack-уведомления делятся на два режима:

- срочно в чат: новый комментарий/вопрос, переход в `Тестирование`, `Нужно уточнение`, задача стала критичной;
- в дневной дайджест: зависшие 24ч+, обычные закрытия, переназначения, спокойные смены статусов и прочий рабочий шум.

Повтор одного и того же срочного события подавляется кулдауном `ATLAS_YOUTRACK_NOTIFICATION_COOLDOWN_MS` (по умолчанию 12 часов), чтобы snapshot-сбой или повторная проверка не спамили чат.

Дневной дайджест отправляется один раз в день в 15:00 по Москве. Он показывает текущее состояние задач, что нужно сделать сейчас, и накопленный рабочий движ за период.

Формат должен оставаться операторским: короткий человеческий вывод сверху, затем конкретные задачи и ссылки. Не писать `Суперсус на связи`: в Telegram и так видно имя и аватарку бота. Не добавлять служебные поля вроде `статус отправлено`, `приоритет` и не упоминать Codex.

Форснуть дайджест для проверки:

```bash
curl -fsS -X POST http://127.0.0.1:8787/api/youtrack/check \
  -H "Content-Type: application/json" \
  -d '{"notify":true,"forceDigest":true}'
```

Для обычных текстовых сообщений в Telegram использовать `/api/telegram/send-message`. Если нужно ответить на конкретное сообщение, передать `replyToMessageId`. Если нужно пометить сообщение реакцией, передать `reactionToMessageId` и `reactionEmoji` или использовать `/api/telegram/react-message`. Для фото/мемов по URL использовать `/api/telegram/send-photo` с `photoUrl`, `caption`, `replyToMessageId` и при необходимости `reactionToMessageId`. Не использовать `/api/telegram/push-subtask` для ручных сводок: он предназначен для служебного push подзадач и добавляет поля ответственного, статуса и приоритета.

Перезапустить:

```bash
sudo systemctl restart atlas-telegram-bot.service
sudo systemctl status atlas-telegram-bot.service
```

Логи:

```bash
sudo journalctl -u atlas-telegram-bot.service -f
sudo journalctl -u hermes-telegram-bridge.service -f
```

Если `TELEGRAM_BOT_TOKEN` не задан, сервис не падает, а работает в disabled-режиме.

## BotFather

Для групповых задач лучше отключить privacy mode:

```text
@BotFather → /setprivacy → выбрать бота → Disable
```

И добавить бота в нужные группы или в отдельный `Atlas Tasks Inbox`.
